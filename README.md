# Essentia Unofficial Fork for Windows (Experimental)

This fork is an **experimental version** of Essentia that aims to provide a more user-friendly installation process, especially for *Windows* users. It includes simplified build instructions to help users get started quickly without the need for complex dependency management.

Linux users are encouraged to use the official repository for a more stable experience.

Please **do not** use this fork for production purposes, as it may contain bugs and is not guaranteed to be stable.

Please **do not** submit issues to the *official* Essentia repository regarding this fork, as it is not maintained by the original developers.

Please **do not** expect pull requests or further development, as it is not officially supported by the Essentia team.

Most of the modification is generated by AI but tested manually.

Please report any issues you encounter.

Tested on Windows 11 + MSYS2/MINGW64 + MinGW GCC 16.1.0 + CPython 3.11 (cp311-win_amd64)

## What Was Changed & Why

This fork modifies the upstream essentia codebase to enable native Windows builds using **MSYS2 + MinGW-w64** (no WSL, no Docker). Below is a summary of every change and its motivation.

### Build System (`wscript`)

| Change | Reason |
|--------|--------|
| Added `_is_mingw()` detection (checks `MSYSTEM` env var or `g++` in PATH) | The stock `wscript` only knows about MSVC on Windows. MinGW needs entirely different compiler flags. |
| `configure()`: new MinGW branch forces `CXX_COMPILER = ['g++']`, sets `CC=gcc`/`CXX=g++` in the environment, and clears `INCLUDE`/`LIB`/`LIBPATH` env vars | waf auto-detects MSVC first; without forcing GCC it picks the wrong toolchain. MSVC environment variables also leak into GCC compilation paths. |
| `configure()`: auto-sets `PKG_CONFIG_PATH` to `$MINGW_PREFIX/lib/pkgconfig` | pkg-config cannot find `.pc` files for ffmpeg, fftw, etc. from the MinGW prefix otherwise. |
| `configure()`: adds `-D_USE_MATH_DEFINES` | MinGW's `<cmath>` does not define `M_PI`, `M_LN2`, etc. without this macro. |
| `configure()`: adds `-static-libgcc -static-libstdc++` | Prevents missing `libgcc_s_seh-1.dll` and `libstdc++-6.dll` at runtime. |
| `configure()`: relaxes SSE flag condition from `sys.platform != 'win32'` to `sys.platform != 'win32' or _is_mingw()` | GCC supports SSE on Windows; previously all Windows platforms were excluded from SSE optimization. |
| `configure()`: relaxed `-pipe` exclusion similarly | `-pipe` is a GCC flag, safe for MinGW but was blocked for all Windows. |
| `options()`: skip MSVC toolchain loading when MinGW is detected | Avoids unnecessary MSVC probing errors. |

### Source Build (`src/wscript`)

| Change | Reason |
|--------|--------|
| Fixed `.pc` file install path: replaced hardcoded `'../build/essentia.pc'` with `ctx.path.get_bld().parent.make_node('essentia.pc')` | The hardcoded path breaks when a custom build output directory is specified via `-o`. |

### Python Bindings Build (`src/python/wscript`)

| Change | Reason |
|--------|--------|
| Replaced `distutils.sysconfig` with `sysconfig` | `distutils` was removed in Python 3.12; this change restores compatibility with modern CPython. |
| Conditionalized `-lpython*` link flag filtering: MinGW now keeps the flag | The original code unconditionally strips `-lpython*` to avoid conflicts on Linux. MinGW needs it to link the CPython library. |

### C++ Fixes — `BOOL` → `ESS_BOOL` (`typedefs.h`, `parsing.cpp`, `pyalgorithm.cpp`, `pystreamingalgorithm.cpp`)

| Change | Reason |
|--------|--------|
| Renamed enum value `BOOL` to `ESS_BOOL` across 4 files (~12 locations) | `<windows.h>` defines `typedef int BOOL`, which conflicts with essentia's `enum Edt { BOOL, ... }`. MinGW includes `<windows.h>` transitively, causing a compilation error. |

### C++ Fixes — NumPy 2.x API (`tensorreal.cpp`)

| Change | Reason |
|--------|--------|
| Replaced `result->strides[3]` with `PyArray_STRIDES((PyArrayObject*)result)[3]` | NumPy 2.x made `strides` a private field; direct access no longer compiles. The accessor macro is the portable API. |

### Python Packaging (`setup.py`)

| Change | Reason |
|--------|--------|
| Added `_is_mingw()` detection function | Enables conditional build flows for Windows/MinGW vs. Linux. |
| MinGW branch skips `build_3rdparty_static_debian.sh` | That script is Linux-only (Debian-specific commands). |
| MinGW config: `--build-static --with-python` (no `--static-dependencies`) | MinGW dependencies come from pacman as shared DLLs, not static libraries. |
| Reads `PYTHONDIR` from `build/c4che/_cache.py` after waf install | On MinGW, waf installs `.pyd`/`.py` directly to `PYTHONDIR` (not `tmp/lib/python*/site-packages/`). We need to locate the output to complete the install. |
| Auto-runs `windows/resolve_dlls.py` after install | Copies transitive DLL dependencies into the package directory so `import essentia` works without manual PATH configuration. |
| Copies `essentia-stubs/` to `site-packages/` | PEP 561 type stubs for IDE autocompletion. |
| `EssentiaInstall` command handles already-installed-to-site-packages case | On MinGW the library ends up in site-packages directly; `shutil.move` would fail without this guard. |
| Added `'Operating System :: Microsoft :: Windows'` trove classifier | Correct metadata for PyPI/package registries. |

### Runtime DLL Loading (`src/python/essentia/__init__.py`)

| Change | Reason |
|--------|--------|
| Added `os.add_dll_directory()` + `PATH` prepending on Windows (`if sys.platform == 'win32'`) | `_essentia.pyd` depends on ~90 transitive DLLs (fftw, ffmpeg, taglib, etc.). Without adding the package directory to the DLL search path, the loader fails with `DLL load failed`. |

### New Files — `windows/` Tooling

| File | Purpose |
|------|---------|
| `windows/resolve_dlls.py` | Recursively scans PE import tables using `objdump -p`, resolves transitive DLL dependencies from the MinGW prefix, and copies them into the essentia package directory. Filters out Windows system DLLs (kernel32, ntdll, etc.). |
| `windows/generate_stubs.py` | Generates PEP 561 `.pyi` type stubs with **exact parameter names, types, defaults, and return types** by scraping `https://essentia.upf.edu/reference/`. Produces `standard.pyi` (251 algorithms), `streaming.pyi` (245 algorithms), and top-level stubs. Results are cached in `doc_cache.json` for fast re-runs. |

### Configuration (`pyproject.toml`)

| Change | Reason |
|--------|--------|
| Updated `numpy` build dependency to `>=2.0.0rc1` | Matches the NumPy 2.x API fix in `tensorreal.cpp`. |
| Added `setuptools >= 48`, `wheel >= 0.29.0` build dependencies | Required for `setup.py`-based builds via pip. |

## Before Setup

```bash
# 1.Prepare (git+scoop+msys2) 

# Use Powershell

cd path/to/project

# uv is optional, you can use python -m venv .venv instead
uv venv --python 3.11 # NOT TESTED WITH PYTHON 3.12 YET
.venv/Scripts/python.exe -m ensurepip

git clone https://github.com/UniDel7a/essentia.git

# PLEASE install scoop to somewhere else rather than on disk C 
# configure scoop to use the main bucket
scoop bucket add main
scoop install main/msys2

# Run msys2 after setup (once)
msys2 

# If msys is still running after initial setup, exit it and run mingw64 shell instead
exit 

# 2. Install dependencies 

# Use mingw64 shell instead of MSYS2 shell, type mingw64 in the command line to open it
mingw64 

# Install dependencies using pacman (once)
pacman -S mingw-w64-x86_64-toolchain
pacman -S mingw-w64-x86_64-pkgconf
pacman -S mingw-w64-x86_64-eigen3
pacman -S mingw-w64-x86_64-fftw
pacman -S mingw-w64-x86_64-ffmpeg
pacman -S mingw-w64-x86_64-taglib
pacman -S mingw-w64-x86_64-libsamplerate
pacman -S mingw-w64-x86_64-libyaml
pacman -S mingw-w64-x86_64-chromaprint

exit
```

## Install (Using pip)

```bash
# 1. Open mingw64 shell instead of MSYS2 shell,activate virtual environment
cd path/to/project
mingw64

# 2. Activate the virtual environment
source .venv/Scripts/activate 

# 3. Install essentia using pip
python -m ensurepip
python -m pip install --upgrade pip setuptools wheel
pip install ./essentia

```

## Install (Build Manually)

```bash
# 1. Open mingw64 shell instead of MSYS2 shell,activate virtual environment
cd path/to/project
mingw64
source .venv/Scripts/activate

# 2. Enter project directory
cd /path/to/essentia

# 3. Configure
# Use mingw64 shell instead of MSYS2 shell
python waf configure --build-static --with-examples --with-python -o build_dir --prefix=../.venv

# 4. Install
python waf install
# Python bindings filename may look like:
# build_dir\src\python\_essentia.cp311-win_amd64.pyd

# 5. Copy dlls to the virtual environment
python windows/resolve_dlls.py

# 7. Generate type stubs
python windows/generate_stubs.py

# 8. Test installation
python -c "import essentia; print(essentia.__version__)"

```

Essentia is an open-source C++ library for audio analysis and audio-based music information retrieval released under the Affero GPLv3 license. It contains an extensive collection of reusable algorithms which implement audio input/output functionality, standard digital signal processing blocks, statistical characterization of data, and a large set of spectral, temporal, tonal and high-level music descriptors. The library is also wrapped in Python and includes a number of predefined executable extractors for the available music descriptors, which facilitates its use for fast prototyping and allows setting up research experiments very rapidly. Furthermore, it includes a Vamp plugin to be used with Sonic Visualiser for visualization purposes. Essentia is designed with a focus on the robustness of the provided music descriptors and is optimized in terms of the computational cost of the algorithms. The provided functionality, specifically the music descriptors included in-the-box and signal processing algorithms, is easily expandable and allows for both research experiments and development of large-scale industrial applications.

Documentation online: http://essentia.upf.edu


Installation
------------

The library is cross-platform and currently supports Linux, macOS, Windows, iOS and Android systems. Read installation instructions:
-  http://essentia.upf.edu/documentation/installing.html 
-  [doc/sphinxdoc/installing.rst](doc/sphinxdoc/installing.rst)

Install from master for the latest updates.

To use in Python (Linux `x86_64`, `i686`): `pip install essentia` or `pip install essentia-tensorflow`.

Docker images: https://hub.docker.com/r/mtgupf/essentia/


You can download and use prebuilt static binaries for a number of Essentia's command-line music extractors instead of installing the complete library
- [doc/sphinxdoc/extractors_out_of_box.rst](doc/sphinxdoc/extractors_out_of_box.rst)

Versions
--------

Official releases: https://github.com/MTG/essentia/releases

Github branches:
- [master](https://github.com/MTG/essentia/tree/master): latest updates; if you got any problem, try it first.

If you use example extractors (located in src/examples), or your own code employing Essentia algorithms to compute descriptors, you should be aware of possible incompatibilities when using different versions of Essentia.
